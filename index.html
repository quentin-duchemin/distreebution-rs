01 Two algorithms, one framework

Trees that optimise proper scoring rules

Standard regression trees split to reduce squared error — they only ever model the mean. These two split on objectives that score the entire predictive distribution, so every leaf carries calibrated uncertainty. Both extend to forests for smoother, stronger forecasts.

PMQRF

Pinball Multi-Quantile RF

forest of Pinball Multi-Quantile Regression Trees

Predicts several quantiles at once by minimising a generalised entropy derived from the Weighted Interval Score — a sum of pinball losses across quantile levels. All quantiles share the same splits, so the forecast stays monotone and interpretable.

split engine: min–max heaps · O(N log N)
CRPS-RF

CRPS Regression Forest

forest of CRPS Regression Trees

Splits by directly minimising the Continuous Ranked Probability Score, a strictly proper scoring rule for full-distribution prediction. It rewards forecasts that are both sharp and well-calibrated — not just centred correctly.

split engine: Fenwick tree · O(N log N)
02 Why a Rust implementation

Same algorithm. Native-code speed.

The split-selection procedures are already O(N log N) per node by design — the min–max heaps and Fenwick tree maintain the order statistics incrementally, so the asymptotics were never the problem. What the Rust port removes is the constant factor: interpreter overhead, per-sample Python object churn, and cache-unfriendly access. The result is a 10–100× wall-clock speedup with bit-identical splits on continuous data.

How training cost grows with d features and N points

For a single tree, fit time is linear in the number of features d (each feature is scanned once per node) and near-linear in N — the log N factor from the incremental order-statistic updates. A CRPS tree on 16 000 points and 50 features fits in ≈1.5 s on one core; a forest simply multiplies by the number of trees and divides by the cores available.

03 The trick behind the speed

Order statistics, maintained incrementally

Evaluating a distributional loss at every candidate split is the bottleneck. Instead of recomputing from scratch, the algorithm slides the split point one sample at a time and updates specialised data structures — the same ones described in the paper.

Min–max heaps

For the pinball / WIS objective, each quantile level keeps a heap of the residuals below it. Inserting the next sorted sample means one heap rebalance — O(log N) — instead of a full re-sort.

Fenwick tree

The CRPS gain reduces to prefix sums over sorted targets. A binary-indexed tree updates and queries those sums in O(log N), turning a quadratic scan into a linearithmic one.

Column-major + rayon

Features are stored contiguously so each candidate scan stays in cache, and the search across features runs in parallel across cores — with the GIL released for the whole fit.

04 Does it actually forecast better

Calibrated and competitive

On data with shifting distributional shape — regions that differ in skew and tail weight, not just in mean — the CRPS forest places splits that mean-squared-error forests miss. Lower CRPS is better; coverage should sit near its nominal level.

Where it shines

Heteroscedastic data — spread that changes across the feature space.
Skewed or heavy-tailed conditionals a Gaussian mean-model would miss.
Tabular problems where you need interpretable calibrated intervals.
Pipelines that pair with conformal prediction for coverage guarantees.

Install & run

Prebuilt wheel — no Rust toolchain needed.

# prebuilt wheel, imports as a native module
pip install distreebu_rs
import distreebu_rs as rs
tree = rs.RegressionTreeCRPS(max_depth=6, min_samples_split=20, loo=True)
tree.fit(X, y)
leaves = tree.get_values_leaf(X_query, idx)

Drop-in shims mirror the original API — swap DisTreebution.CRPSRT for DisTreebution_rs.CRPSRT and keep the rest of your code.
